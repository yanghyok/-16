# 한양혁 AX 제안서

## 현재 업무 흐름

시트에서 확인된 주요 업무 축은 다음과 같습니다.

1. **서버 및 백업 점검** — 출근 후 서버 관리 프로그램으로 서버·백업 상태를 매일 점검하고, 이상 발생 시 처리 결과를 팀원과 공유하거나 외부 관리 업체에 문의합니다. 일반적이지 않은 에러의 원인 파악과 외부 업체와의 커뮤니케이션에 시간과 부담이 집중됩니다.
2. **개발 요청 분석 및 요구사항 구체화** — 요청 부서로부터 개발 요청서를 수신한 뒤, 인터뷰를 반복하며 추상적인 요청을 실제 업무 흐름 기준으로 구체화하고 설계서를 작성합니다. 요청 내용이 모호한 경우가 많아 인터뷰 횟수가 늘어나는 것이 주된 부담입니다.

두 업무 모두 '비정형 텍스트(점검 로그 또는 요청서)를 구조화하여 공유·보고 가능한 산출물로 만든다'는 구조를 공유합니다.

---

## 단계 0: 환경 준비

아래 항목은 미니 단계를 시작하기 전에 먼저 확인·준비가 필요합니다.

- **점검 로그 텍스트 확보 경로**: 서버 관리 프로그램에서 점검 결과나 에러 메시지를 텍스트로 복사할 수 있는지 확인합니다. Claude Code와 서버 관리 프로그램 간의 직접 연동 가능 여부는 사내 도구 정책에 따라 달라지므로, 우선 텍스트 붙여넣기 방식으로 시작합니다.
- **샘플 점검 로그**: 최근 이상이 발생했던 에러 메시지 또는 점검 결과 텍스트 3~5건을 준비합니다 (분류 기준 학습용).
- **개발 요청서 샘플**: 과거에 처리했던 개발 요청서 3~5건(요청 내용과 최종 확정된 요구사항이 함께 있는 것이면 더 좋습니다)을 텍스트 또는 파일로 준비합니다.
- **외부 업체 커뮤니케이션 샘플**: 외부 관리 업체에 문의했던 내용(에러 설명 + 문의 메시지) 2~3건을 준비합니다 (문의 초안 생성 학습용).

---

## 추천 접근 방식

스킬을 먼저 설계하지 않습니다. Claude Code에 자연어로 의도를 전달하여 작업을 먼저 실행시키고, 결과가 만족스러운 수준에 도달하면 그 작업을 스킬로 저장합니다.

```
의도 전달 → 결과 확인 → 조정 → 만족 → 스킬화 → 반복 사용 → 업데이트
```

스킬의 구조 설계는 Claude Code가 담당하므로, 담당자는 "무엇을 하고 싶은지"만 명확히 전달하면 됩니다. 한 번의 의도에 여러 조작을 뭉치지 않고, 한 단계씩 의도를 좁혀 전달하며 결과를 확인하는 흐름이 핵심입니다.

---

## 시작 의도 문장

업무별로 아래 순서대로 진행합니다. 각 단계는 이전 단계 결과를 사람 확인으로 점검한 뒤 다음 단계로 넘어갑니다. 의도 문장은 실제 연결 상태·데이터 규모·필터 기준에 따라 조정이 필요한 초안입니다.

---

### ① 서버 점검 결과 정리 및 공유 초안 생성 — 가칭

**단계 1: 최소 입력으로 첫 산출**

> "아래 텍스트는 오늘 서버 점검에서 나온 에러 메시지입니다. 이 메시지가 '정상 범위 내 경고'인지 '즉시 처리가 필요한 장애'인지 판단해 주세요. (에러 메시지 붙여넣기)"

_(사람 확인: 판단 결과가 실제 경험과 맞는가, 오분류된 항목이 있는가.)_

**단계 2: 범위 확장**

> "오늘 점검에서 확인한 항목 전체 목록을 붙여넣겠습니다. 각 항목을 '정상 / 경고 / 즉시 처리 필요' 세 상태로 분류해 주세요."

_(사람 확인: 항목 수가 빠짐없이 반영되었는가, 분류 기준이 실제 판단과 일치하는가.)_

**단계 3: 이상 항목 필터링**

> "이 목록에서 '즉시 처리 필요'로 분류된 항목만 남겨 주세요."

_(사람 확인: 실제 오늘 처리해야 할 항목이 맞는가.)_

**단계 4: 원인 후보 탐색**

> "이 에러 메시지의 일반적인 원인 후보를 두세 가지 제안해 주세요. 각 후보 옆에 확인 방법도 한 줄씩 적어 주세요."

_(사람 확인: 제안된 원인 후보가 실제 환경에서 발생 가능한 것들인가, 빠진 후보가 있는가.)_

**단계 5: 외부 업체 문의 초안 작성**

> "처리하지 못한 항목에 대해 외부 관리 업체에 전달할 문의 메시지 초안을 작성해 주세요. 에러 메시지와 현상 설명을 포함해 주세요."

_(사람 확인: 설명이 충분히 구체적인가, 빠진 정보가 없는가.)_

**단계 6: 팀원 공유 메시지 포맷팅**

> "오늘 점검 결과를 팀원에게 공유할 수 있는 요약 메시지로 정리해 주세요. 상태별로 구분하고 처리 완료 항목과 미처리 항목을 구별해 주세요."

_(사람 확인: 팀원이 한눈에 파악하기 좋은 형태인가, 톤이 적절한가.)_

**단계 7: 반복 조정**

> "미처리 항목 앞에 '확인 필요'라는 표시를 추가하고, 외부 업체 문의 여부도 한 줄로 포함해 주세요."

_(사람 확인: 이 형태로 매일 공유해도 될 수준인가.)_

**단계 8: 스킬화**

> "이 흐름을 '서버 점검 결과 정리 및 공유' 스킬로 저장해 주세요. 분류 기준(정상/경고/즉시 처리 필요)과 공유 메시지 포맷을 스킬 안에 고정해 두고, 새로운 에러 유형이 생기면 추가할 수 있게 해 주세요."

---

### ② 개발 요청 요구사항 구체화 보조 — 가칭

**단계 1: 최소 입력으로 첫 산출**

> "아래는 요청 부서에서 받은 개발 요청서 내용입니다. 이 요청에서 아직 불명확한 부분이 무엇인지 짚어 주세요. (요청서 내용 붙여넣기)"

_(사람 확인: 지적된 불명확 항목이 실제로 인터뷰에서 자주 막히는 지점과 일치하는가.)_

**단계 2: 인터뷰 질문 목록 생성**

> "이 불명확한 항목들을 해소하기 위해 요청자에게 물어볼 질문 목록을 만들어 주세요. 업무 흐름 순서대로 정렬해 주세요."

_(사람 확인: 질문이 요청자가 답하기 쉬운 수준인가, 누락된 핵심 질문이 있는가.)_

**단계 3: 인터뷰 답변 반영**

> "아래는 요청자와 나눈 인터뷰 메모입니다. 이 내용을 반영하여 요구사항 항목을 구체화해 주세요. (인터뷰 메모 붙여넣기)"

_(사람 확인: 답변이 제대로 반영되었는가, 아직 남은 불명확 항목이 있는가.)_

**단계 4: 요구사항 분류 탐색**

> "구체화된 요구사항을 '핵심 기능 / 부가 기능 / 추후 검토'로 나눌 수 있는지 각 항목에 어느 유형이 맞을지 제안해 주세요."

_(사람 확인: 분류가 실제 개발 우선순위와 맞는가, 조정할 항목이 있는가.)_

**단계 5: 분류 확정 및 요구사항 목록 정리**

> "조정된 분류 기준으로 요구사항 목록을 최종 정리해 주세요."

_(사람 확인: 팀장·임원 보고 전에 빠진 항목이 없는가.)_

**단계 6: 보고용 요약 포맷팅**

> "이 요구사항 목록을 팀장 및 임원 보고에 쓸 수 있는 한 장 요약으로 정리해 주세요. 요청 배경, 핵심 기능, 예상 작업 범위 순서로 구성해 주세요."

_(사람 확인: 보고 자리에서 설명하기에 충분한 내용인가, 불필요하게 긴 부분이 있는가.)_

**단계 7: 반복 조정**

> "핵심 기능 항목마다 '요청자 확인 완료 여부'를 한 줄로 표시해 주세요."

_(사람 확인: 이 형태로 보고서를 제출해도 될 수준인가.)_

**단계 8: 스킬화**

> "이 흐름을 '개발 요청 요구사항 구체화' 스킬로 저장해 주세요. 질문 목록 생성 기준과 보고 요약 포맷을 스킬 안에 고정해 두고, 요청 유형이 달라지면 기준을 추가할 수 있게 해 주세요."

## 스킬 업데이트 방식

만들어진 스킬은 실무 과정에서 반복 호출하며 개선합니다. 엉뚱하거나 부정확한 응답이 발생한 시점에 다음과 같이 전달합니다.

> "이 경우는 이렇게 처리되어야 합니다. 스킬에 반영해 주세요."

새로운 에러 유형이나 요청 패턴이 등장하면 추가하고, 반복되는 실수 패턴은 금지 규칙으로 박제하는 방식으로 스킬이 점진적으로 정교해집니다. 주 1회 10~15분 정도 정리 시점을 정해두면 한 달 안에 실무 대응 가능한 수준에 도달합니다.

---

## 스킬 간 연결 구조

각 스킬이 일정 수준에 도달하면 다음 흐름으로 이어집니다.

**흐름 1 — 서버 점검 결과 처리**
```
에러 메시지 텍스트 붙여넣기 → [서버 점검 결과 정리 스킬] → (사람 확인: 분류 점검) → 팀원 공유 메시지 산출
                                                              ↓
                                               [외부 업체 문의 초안] → (사람 확인: 내용 검토) → 문의 발송
```

**흐름 2 — 개발 요청 처리**
```
개발 요청서 텍스트 붙여넣기 → [개발 요청 구체화 스킬] → (사람 확인: 질문 점검) → 인터뷰 진행
                                                          ↓
                                           인터뷰 메모 반영 → (사람 확인: 요구사항 확인) → 보고용 요약 산출
```

두 흐름 모두 '비정형 텍스트 붙여넣기 → AI 구조화 → 사람 확인 → 산출물 완성' 패턴을 공유합니다. 텍스트를 구조화하는 첫 단계가 두 스킬 모두에 재사용됩니다.

처음부터 이 그림을 완성하려 하지 마시고, 각 스킬을 하나씩 만들고 깎아가시다 보면 이 연결이 자연스럽게 드러납니다.
